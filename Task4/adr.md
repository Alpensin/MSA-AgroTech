### <a name="_qmphm5d6rvi3"></a>**Критерии выбора решений**

Основным критерием выбора между двумя решениями является автономность фермы и удобство для дальнейшего развития продукта.

Сравнение по критериям:
|**№**|**Требование**|**Решение main**|**Решение alternative**|
| :-: |:-: |:-: | :- |
|1|Задержка до 10 минут без учета проблем со связью|Да|Выполняется при обеспечении канала связи от 2 МБит/с на камеру на ферме.
|2|Резерв связи для отправки уведомлений оператору|Да|Да|
|3|Возможность интегрировать заказчикам при реализации как SAAS|Да. Требует больше ресурсов для внедрения|Да|
|4|Работа автономно при потере связи с Центральным Узлом|Да|Частично. Пропадает видеоналитика|
|5|Иметь API для мобильного приложения и веб-приложения|*Да|*Да|
|6|Позволять системе видеоаналитики реагировать в реальном времени (миллисекунды)|Да|**Да|
|7|Иметь высокую производительность — от момента возникновения нештатной ситуации, зафиксированной с помощью видеоаналитики, должно проходить не более 5 секунд до момента оповещения|Да|**Да|
|8|Отказоустойчивость 99,95%|Да|**Да|
|9|Быть расширяемой, то есть иметь возможность разработать новый функционал без изменений существующего|Да, но сложнее обеспечить расширение видеоаналитики|Да

_*Оба решения предназначены для MVP. Подразумевается выделение API-Gateway как для ферм, так и для центрального узла. Также планируется внедрение авторизации._
_**В случае достаточной пропускной способности канала между фермой и центральным узлом_


Для поддежания максимальной автономности рекомендуется оставить сервис расписаний в контуре фермы. Это позволит управлять фермой полностью автономно в случае отсутствия связи.

Альтернативное решение может лучше подходить для SAAS решений, т.к. потребует более простой интеграции на ферме заказчика. 

Для MVP рекомендуется остановиться на основном решении. В дальнейшем можно подключать фермы по обоим сценариям, в зависимости от размера ферм, качества Интернета на фермах и тарифов.
